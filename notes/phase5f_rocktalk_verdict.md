# Phase 5f — RockTalk-weights × our-pipeline empirical test

**Date:** 2026-05-21
**Status:** **DECISIVE — bug is in our pipeline forward-pass, NOT our converter.**
**Outcome:** byte-identical pixel output (V0_baseline.md5 ≡ RT_through_our_pipeline.md5 = `2ca49d9007cee9f22fdfd124c94fd53f`).

## TL;DR

Downloaded RockTalk/Lance-3B-Video-MLX (26 GB F32 safetensors). Remapped their
keys to our LanceModel layout (`scripts/22_remap_rocktalk.py`), cast F32 →
bf16 to match our converter's behavior. Ran the remapped weights through
**OUR** pipeline at the exact same config as V0_baseline (256²×17f, 30 steps,
CFG=4.0, seed=42, MaPE=None, sms=1, rope_fp32=False). The midframe PNG and
MP4 are **byte-identical** to V0_baseline.

Combined with the fact that RockTalk's HF card ships **sharp** ocean-wave
T2V samples at 256²×9f produced by *their* pipeline against *the same
weights* (visible on `huggingface.co/RockTalk/Lance-3B-Video-MLX`), this
proves:

1. **Our converter is correct.** RockTalk's F32 weights cast to bf16 produce
   numerically equivalent values to ours — both derive from the same
   upstream PyTorch checkpoint, and bf16 truncates the same way.
2. **The bug is in OUR pipeline code.** Same weights + their pipeline =
   sharp; same weights + our pipeline = blurry painterly. The difference
   must be in the forward-pass logic, not the weight values.

This dramatically narrows the search space. **Stop investigating the
converter, the VAE keys, and bf16 precision.** The painterly aesthetic is
generated by a bug somewhere in `pipeline/t2v.py` or `model/lance_llm.py`.

### Phase 5f.1 — Remap (~17s)

`scripts/22_remap_rocktalk.py` translates RockTalk's keys to our layout:
- Drop `language_model.{model.}?` prefix
- `time_embedder.fc1/fc2` → `time_embedder.proj_in/proj_out`
- `vae2llm.*` → `vae_in_proj.vae2llm.*`
- Skip 360 `vit_model.*` keys (we use separate vit.safetensors)
- Cast F32 → bf16 for non-norm tensors

Output: `Lance-3B-Video-bf16-RT/model.safetensors` (12 GB bf16, 1051 tensors).
Independent file from our Phase 1b converter output (different md5s, both
12 GB).

## Implications for the bug hunt

### What's now off the table

- ✅ **Our converter is correct.** Don't touch `scripts/02_convert.py`.
- ✅ **bf16 precision is sufficient.** RockTalk's bf16-precision weights
  produce sharp output through their pipeline; bf16 is not the bottleneck.
- ✅ **P0a (fp32 RoPE) is a no-op.** Phase 5g matrix proved byte-identical
  output regardless of fp32 vs bf16 RoPE rotation.
- ✅ **P0b (sms=2) actively hurts.** Phase 5g matrix proved subject is
  lost at sms=2.
- ✅ **MaPE shift is correctly disabled.** Phase 5d default `mape_anchor=None`
  is the right choice (no Phase 5f re-test needed).
- ✅ **The Wan2.2 VAE in our pipeline is correct enough.** RT used our VAE
  for this test and the output is still equivalent. (Their VAE may be
  BETTER for the decode step alone, but it's not the cause of the painterly
  *aesthetic* — that's pre-decode.)

### What's left to investigate

The bug must be in OUR PIPELINE FORWARD-PASS CODE, in one of:

1. **CFG composition** (`t2v.py` generate loop, lines around the `velocity`
   computation). We do three-component CFG with cfg_renorm_type='global'
   — verify exactly matches upstream's `cfg_text_unconditional_pro_new`.

2. **Euler sampler step** (`t2v.py` `latents = latents + dt * velocity`).
   Verify timestep_shift application order matches upstream.

3. **TimestepEmbedder input domain** (research P2 — `t * 1000`?). Our
   `time_embedder(t.reshape(1))` — what range does t span? Upstream uses
   sigma ∈ [0, 1] but TimestepEmbedder might internally expect t * 1000.

4. **Latent token block mask** (`_build_block_mask`). We made it
   bidirectional within the latent block per Phase 3e. Verify still
   correct for video (3D grid, not just 2D image).

5. **Chat template / prompt assembly** (`_prepare_state`). Our wrapper:
   ```
   <|im_start|>system\n{instruction}<|im_end|>\n
   <|im_start|>user\n{prompt}<|im_end|>\n
   <|im_start|>assistant\n
   <|vision_start|>{video_pad*n_lat}<|vision_end|>
   ```
   Compare against RockTalk's tokenization output for the same prompt.

6. **VAE input projection / flow head** (`vae_in_proj`, `llm2vae`).
   Layer norms / activations / additive position embedding order matters.

7. **VAE-latent normalization** at encode/decode time. Lance's VAE has a
   z_std/z_mean we might not be applying.

### Next step: P2 (TimestepEmbedder t*1000 audit)

Research's P2 candidate is one line and is independent of the failing P0a/
P0b: verify whether our `time_embedder(t)` sees `t ∈ [0, 1]` (sigma) or
`t ∈ [0, 1000]` (legacy scaled). Cheap to test.

After P2, the next leverage is to **clone RockTalk's `lance_mlx`
repository** and diff their forward-pass code against ours. That gives
us the exact pipeline-level diff that produces sharp vs blurry output
— much more direct than another round of hypothesis-driven candidates.
